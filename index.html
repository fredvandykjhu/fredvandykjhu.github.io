Parallel Databases:  Data Collection and Data management

[a]

Why use separate databases for Data Collection and Data management?

The Open Data Kit Data Collection (ODK-DC)  database is the datastore automatically generated in MySQL by the ODK Aggregate platform.  The SMSS Data Management database is a custom database developed by the PREVNAME team to be the primary source for Data Management (Clusters, Users, Data Collectors, Death and Pregnancy tracking, GPS and mapping).  

Raw data from the ODK forms is stored in the ODK-DC database.  ODK systematically creates individual groups of tables based on form configurations.  For example, the Event form.  In this form data is split across two or three tables (Example EV1 group).    ODK documentation recommends that external programming code not be designed to connect directly to these tables.    This is especially important in a  long term or longitudinal project. Data collection forms are expected to evolve over time and each new version of a form will create a separate group of tables (example EV2 group)

 It is possible to add custom tables and other objects into the same database (ODK-DC), but having a separate database for Data Management is preferable. For PREVNAME, the two databases sit side-by-side on the same server.  The modularity simplifies restoring or rebuilding either database.  If needed for data validation, queries can be written that join tables across the two databases.

For the prevname system we created triggers on the key ODK-DC tables.  Whenever a new record is inserted, the record is copied to the Event table in the SMSS database.   Thus, several groups of ODK Event tables (EV1, EV2, etc.)  can be merged into a single Event table in the SMSS database.   Currently PREVNAME ODK is using a 4th version of the Event form.

The SMSS database has a web interface described below as the Dashboard or Operational Dashboard.

Why were 3 machines used?

From the beginning, there was a clear mandate to protect participant privacy and keep identifiable data separate from data that is shared with the public.   Thus, it was important that the public site be hosted on a machine with physical separation from Data Collection and Analysis.   This part of the system was designed to minimize the danger of accidental exposure of protected data.  

The public website only hosts de-identified data and thus it was acceptable to host this on a commercial platform (wordpress.com).  

The Data Analysis Portal was originally developed on a separate machine because the underlying software was more suited to a different type and version of operating system, web server, and database.   It was also felt that the performance of the data collection server could be affected by long and complicated algorithms occasionally run on the data analysis portal.  And vice versa,  at the end of a workday, when many data collectors would upload data, this could affect performance of the Analysis Portal.    

For newer deployments, it would be possible to combine the Data Analysis and Collection websites onto a single machine.  The newer version of the  Data Analysis Portal is docker-based and platform neutral.   A single machine could be adequately provisioned and managed to ensure adequate resources of memory and processing power are readily available for Data Collection and Data Analysis.

 

Cloud server vs. Local server

The key issues are up-time, privacy, flexibility, and cost..    

Up-time: A cloud-based server on a large platform should provide 99.9% uptime.  It will be very reliable. There will be basically no concerns about power outages or internet failures at your server site.  

Privacy:  This can be a sticking point for many projects where there is a desire or need for the data to physically remain in the country where data is being collected.  It is important to be very careful about legal and ethical requirements for participant privacy.   Data encryption during transmission and at rest is  a core principle in securing the data.

Flexibility:  It can be challenging to correctly estimate the specifications of a server before data collection has begun.  And needs can change over time.   With most cloud machines, RAM, disk space, processing power can all be increased and decreased as needed.

Cost: It is important to watch costs when configuring a cloud server, but even without considering extra local server costs for electricity, internet service, fixed IP’s, hours of technical support,  “Renting” a cloud server for 3 years is often significantly cheaper than purchasing a physical machine.

CRVS

MZ Civil Registration and Vital Events system (CRVS)

Integration with the Mozambique CRVS system is being piloted in a selection of clusters in Inhambane.   A custom ODK event form was created with consultation from the Mozambique Ministry of Justice (MOJ).   The adapted event form contains all basic PREVNAME questions and additional fields designed to capture the complete information needed to notify the MOJ CRVS system that a birth or death has occurred.  

A Short Message Service (SMS) system was piloted that allowed the data collector to report a birth or death via a series of text messages.  When successfully used, the data collector would receive a registration number that the family could use to facilitate registration at the nearest CRVS office or post.  

Birth and Death event data from these clusters are shared with MOJ to demonstrate the possibility of further integration between the NEW-NAME and the CRVS systems.

DHIS-2

DHIS2 system

A DHIS2 instance was developed with a plan to eventually incorporate the instance into the MZ MOH DHIS instance.  

Datasets for the Mozambique  DHIS2 instance are prepared on the Data Analysis Portal.  The data is reshaped to match the DHIS2 specifications and then exported as CSV files.  These datasets are posted to the DHIS2 server and imported cyclically.

The data teams followed an extensive process to organize the data and provide a concise and useful interface for integration with the existing Ministry of Health DHIS system.   The NEW-NAME DHIS system has 6 sections:

  1. About NEW-NAME data
  2. Population Summary (shown)
  3. Births Summary
  4. Deaths Summary
  5. Mortality Rates
  6. Cause of Death

All sections have both National and Provincial level data summaries.

Discussion of System Strengths

  • Scalability and Adaptability: At the beginning of the project, many parameters were unknown or expected to change.   The system proved to be dynamic and easily adaptable as the project became more defined.
  • Common software tools kept software costs relatively low.  No external IT consultants needed.  Most software tools do not require licensing fee or licenses were covered by existing institutional licenses.
  • Shared Analysis Portal
  •  Daily Synchronization with data collected in field
  • Great for analyst collaboration
  • Many good features that contributed to project success
  • Programmatic access to data store.   In NEW-NAME, this was provided by direct SQL database access.  If using the newer ODK Central platform or other data collection platforms, this could be provided by an API.
  • Generation of adjacent databases and data objects

Discussion of System Weaknesses

  • Capturing consent signature:    High impact to ODK upload performance.  Significantly increased database size.
  • Signature was captured using ODK signature widget.    This allows the participant to draw their signature on the tablet.    One issue we did not anticipate is that this creates an image with a much larger file than is created by the normal text recorded within a form.  The image is submitted to the server form storage.    The increased size of the data submitted was a heavy burden on the network usage,  increased write times on the server, and lead to much larger databases.   The exact scale of the increase is not carefully documented, but it is estimated at around 10x data.    During study startup when we were enrolling thousands of households per day.   Many data collectors would submit all their data in the evening.   Data collectors had many occasions where the upload was very slow and sometimes they would lose the connection.     There was some bottleneck at the server, but mainly the problem seemed to be very slow cell phone data in areas with only 3G service.
  • Household Identifiers (HHID):     Many errors in initial assignment.  Physical labeling of houses largely unsuccessful.   Difficulty in connecting events (pregnancy, birth, death) to HHID.
  • Overview: HHID’s were assigned sequentially within each Cluster from 001 to n.   The target for n was 300.    Because ODK is a one-way system (data is sent to the server but is not returned to the data collector) supporting systems were used to track the HHID assignment.  
  • Initial assignment:  CSA’s began enrolling households using GPS polygon overlays to determine their location.   They submitted household forms and assigned the HHID sequentially as they progressed.    
  • Mapping team:  In Phase I provinces, the cartography team began field visits to map households after the CSA’s had begun data collection.   Some clusters had to be corrected because of differences in location between cartography HH mapping and CSA work areas.
  • Physical assignment: In some areas, household members objected to having a physical label or written HHID drawn on their home.    In these cases, no physical ID was applied the the home.
  • Summary:   It can be very challenging to correctly assign and maintain households and members.  Families move.  Individuals within households move in and move out.   For the purposes of mortality at a provincial and national level, it may not be necessary to have a strong association between individual death events and precise home of the deceased.    It may be enough to have a strong census and cyclical updated population within each cluster.
  • ODK system captures discrete offline events. This can lead to several issues:
  • Duplicates
  • Difficulty connecting Death event and VASA follow up
  • Invalid DeathID or Wrong DeathID
  • No edit capability
  • CSA’s do not have access to dashboard on ODK Aggregate.   They do not have direct feedback about the data they have collected.  They cannot see previously submitted data.
  •   Difficulty supporting remote data collectors:   The most remote data collectors are the most difficult to support.  They are very important because they are monitoring remote communities  that generally have poor access to resources.   They often have limited internet and cell phone availability, it is difficult to call or text them,   and it is difficult to visit them in person.  
  • Updating data collection settings and form version – new versions of mobile platforms automate form versioning
  • Study Procedure updates
  • Support attempted via:
  • Phone or text communication
  • Remote control of phone (TeamViewer App)
  • In person
